# Glossary

ID types, data relationships, and key concepts across the Telegram packages.

---

## ID Types

| Term | What it is | Where it comes from | Example |
|------|-----------|---------------------|---------|
| `accountId` | MongoDB `_id` of a TelegramAccount document | Created when you `POST /accounts` | `"507f1f77bcf86cd799439011"` |
| `chatId` | Telegram's numeric chat/user ID, stored as a string | From Telegram API | `"123456789"` |
| `conversationId` | Same as `chatId` -- the inbox uses this as the conversation grouping key | Derived from Telegram chat ID | `"123456789"` |
| `contactId` | Your system's internal ID for a person (can be your CRM's contact ID) | You assign it when creating an identifier | `"crm-contact-42"` |
| `identifierId` | MongoDB `_id` of a TelegramIdentifier document | Created when you `POST /identifiers` | `"507f1f77bcf86cd799439012"` |
| `telegramUserId` | Telegram's numeric user ID (same as `chatId` for DMs) | From Telegram API | `"123456789"` |
| `messageId` | Telegram's message ID within a chat, stored as a string | From Telegram API | `"4521"` |
| `runId` | Unique ID identifying a rule engine run | Generated by `POST /runner/trigger` | `"run_a1b2c3d4e5f6"` |
| `ruleId` | MongoDB `_id` of a rule document | Created when you `POST /rules` | `"665def789abc123456000001"` |
| `templateId` | MongoDB `_id` of a template document | Created when you `POST /templates` | `"665abc123def456789000001"` |
| `sessionId` | MongoDB `_id` of a ConversationSession document | Created automatically when inbox tracks a conversation | `"507f1f77bcf86cd799439013"` |

---

## Key Relationships

- **In DMs**, `chatId` = `telegramUserId` = `conversationId` -- they're all the same number, just the Telegram user ID of the other person.
- **`contactId` is YOUR ID** -- map it however you want to your CRM. It's the foreign key that connects the Telegram world to your system.
- **`identifierId` is the library's internal ID** for a TelegramIdentifier record. You rarely use this directly; the rule engine resolves it through your `findIdentifier` adapter.
- **`accountId` identifies which Telegram account** (phone number) you're operating with. Every send, every message, every connection is scoped to an account.
- **`conversationId` ties inbox messages together.** All messages in the same chat share the same `conversationId`, regardless of which account sent them.

## Identifier Fields

A TelegramIdentifier record has these fields:

| Field | Type | Description |
|-------|------|-------------|
| `_id` | ObjectId | The `identifierId` |
| `contactId` | string | Your CRM's contact ID (required) |
| `telegramUserId` | string | Telegram user ID (required, unique) |
| `username` | string | Telegram @username (optional) |
| `firstName` | string | First name (optional) |
| `lastName` | string | Last name (optional) |
| `phone` | string | Phone number (optional) |
| `status` | string | `active`, `blocked`, `privacy_blocked`, `inactive`, `invalid` |
| `knownByAccounts` | ObjectId[] | Which of your accounts have interacted with this contact |
| `sentCount` | number | Total messages sent to this contact |
| `lastActiveAt` | Date | Last activity timestamp |

## Message Fields

A TelegramMessage record has these fields:

| Field | Type | Description |
|-------|------|-------------|
| `accountId` | string | Which of your accounts is involved |
| `conversationId` | string | The chat ID this message belongs to |
| `messageId` | string | Telegram's message ID (unique) |
| `senderId` | string | Who sent it (Telegram user ID or your account ID) |
| `senderType` | string | `account` (your account sent it) or `user` (the other person sent it) |
| `direction` | string | `inbound` (received) or `outbound` (sent) |
| `contentType` | string | `text`, `photo`, `video`, `voice`, `audio`, `document`, `sticker`, `location`, `contact` |
| `content` | string | The message text (or caption for media) |
| `mediaType` | string | Media type if applicable |
| `mediaUrl` | string | Media URL if applicable |
| `readAt` | Date | When the message was marked as read (null if unread) |

## Conversation Session Fields

A ConversationSession tracks the state of an ongoing conversation:

| Field | Type | Description |
|-------|------|-------------|
| `accountId` | string | Which of your accounts is in this conversation |
| `contactId` | string | Your CRM's contact ID |
| `identifierId` | string | Reference to the TelegramIdentifier (optional) |
| `conversationId` | string | The chat ID |
| `status` | string | `active`, `paused`, `closed` |
| `startedAt` | Date | When the session began |
| `endedAt` | Date | When the session was closed (null if active) |
| `messageCount` | number | Messages exchanged in this session |
| `lastMessageAt` | Date | Timestamp of the last message |

## Account Statuses

| Status | Meaning |
|--------|---------|
| `connected` | TDLib client is active, account can send and receive messages |
| `disconnected` | Client is not running, account is idle |
| `warmup` | Account is in the warmup phase, sending with reduced limits |
| `quarantined` | Account is temporarily disabled due to errors (auto-releases after cooldown) |
| `banned` | Telegram has permanently banned this account |
| `error` | Connection or auth error occurred |

## Key Concepts

### TDLib vs Bot API

Two completely different ways to interact with Telegram:

- **TDLib** (Telegram Database Library) -- lets you operate as a **user account**. Messages appear as coming from a real person. Requires phone number auth + OTP. The `account-manager` package uses TDLib via GramJS.
- **Bot API** -- lets you operate as a **bot account**. Messages show the bot's name and a "BOT" badge. Requires a bot token from @BotFather. The `bot` package uses the Bot API.

They have completely different auth systems, different rate limits, and different capabilities. TDLib can do things bots can't (join groups as a user, see full chat history, appear as a real person). Bots have features TDLib doesn't (inline keyboards, payments, webhook mode).

The two can work together: use the bot for admin commands and notifications, use TDLib accounts for actual outreach.

### Session String

A serialized TDLib auth session. Think of it as a "remember me" token for a Telegram user account.

**How it works:**
1. You call `POST /sessions/request-code` with a phone number -- Telegram sends an OTP
2. You call `POST /sessions/verify-code` with the OTP -- the library authenticates and returns a session string
3. You store that session string in the account record via `POST /accounts`
4. Every time the account connects, it reuses the session string instead of re-authenticating

Session strings are sensitive credentials. They're redacted in API responses and should be treated like passwords.

### Warmup

New or dormant Telegram accounts that suddenly send many messages get flagged and banned. Warmup is a built-in system that gradually increases the daily send limit over ~2 weeks.

**Default warmup schedule:**

| Days | Daily Limit | Delay Between Sends |
|------|------------|-------------------|
| 1--3 | 10 messages | 60--120 seconds |
| 4--7 | 25 messages | 45--90 seconds |
| 8--14 | 50 messages | 30--60 seconds |
| 15+ | 100 messages | 15--45 seconds |

During warmup, the account's status is `warmup` instead of `connected`. It can still send messages, but the capacity is limited. Warmup advances automatically (one day per 24 hours) if `autoAdvance` is enabled.

You can customize the schedule in config or skip warmup entirely by setting `options.warmup.enabled: false`.

### Quarantine

When Telegram returns `PEER_FLOOD` or `USER_RESTRICTED` errors, the account is automatically quarantined to protect it from a permanent ban.

**What happens during quarantine:**
1. Account status is set to `quarantined`
2. TDLib client is disconnected
3. A `quarantinedUntil` timestamp is set (default: 24 hours from now)
4. The `onAccountQuarantined` hook fires
5. A background monitor checks periodically and auto-releases accounts once the quarantine expires
6. After release, the account status returns to `disconnected` (you need to reconnect it)

You can also manually quarantine or release accounts via `POST /accounts/:id/quarantine` and `POST /accounts/:id/release`.

### Health Score

A 0--100 score tracking account reliability. Every account starts at 100.

**How it changes:**

| Event | Score Change |
|-------|-------------|
| Successful send | +2 (capped at 100) |
| Send error | -10 |
| Flood wait error (`PEER_FLOOD`) | -20 |

Consecutive errors are tracked separately. When health drops below the critical threshold (20), the account is considered unreliable.

The `selectAccount` adapter in the rule engine can use health scores to pick accounts -- the `highest-health` rotation strategy always picks the healthiest account with remaining capacity.

**Monitor health:**

```bash
# All accounts
curl http://localhost:3000/api/accounts/accounts/health

# Single account
curl http://localhost:3000/api/accounts/accounts/507f1f77bcf86cd799439011/health
```

### Account Rotation

When the rule engine needs to send a message, it calls your `selectAccount` adapter to pick which account sends it. The built-in `AccountRotator` supports three strategies:

| Strategy | Picks by | Best for |
|----------|---------|----------|
| `highest-health` | Highest health score with remaining capacity | Safety -- avoids risky accounts |
| `round-robin` | Cycles through accounts sequentially | Even distribution |
| `least-used` | Lowest usage percentage today | Maximizing total capacity |

### Throttle

Per-recipient rate limiting in the rule engine. Prevents spamming individual contacts.

| Setting | Default | Meaning |
|---------|---------|---------|
| `maxPerUserPerDay` | 1 | Max messages to one contact per day |
| `maxPerUserPerWeek` | 2 | Max messages to one contact per week |
| `minGapDays` | 3 | Minimum days between messages to the same contact |
| `throttleWindow` | `rolling` | `rolling` = sliding window, `fixed` = calendar-based |

These are system-wide settings that apply across all rules. A contact that was already messaged today by Rule A won't be messaged again by Rule B if `maxPerUserPerDay` is 1.

### Error Categories

The rule engine classifies send errors into categories:

| Category | Errors | What happens |
|----------|--------|-------------|
| **critical** | `AUTH_KEY_UNREGISTERED`, `SESSION_REVOKED`, `USER_DEACTIVATED_BAN`, `PHONE_NUMBER_BANNED` | Account is permanently unusable |
| **quarantine** | `PEER_FLOOD`, `USER_RESTRICTED` | Account is quarantined (temporary cooldown) |
| **skip** | `USER_NOT_FOUND`, `USER_PRIVACY_RESTRICTED`, `USER_IS_BLOCKED`, etc. | This contact can't be reached, skip and continue |
| **recoverable** | `TIMEOUT`, `NETWORK_ERROR`, `RPC_TIMEOUT`, `CONNECTION_ERROR` | Temporary network issues, may retry |

### Capacity

Each account has a daily send limit (`currentDailyLimit`). Capacity = how many more messages the account can send today.

```bash
# Check all accounts' remaining capacity
curl http://localhost:3000/api/accounts/accounts/capacity
```

Response includes `dailyMax`, `sentToday`, `remaining`, and `usagePercent` per account, plus `totalRemaining` across all connected accounts.
