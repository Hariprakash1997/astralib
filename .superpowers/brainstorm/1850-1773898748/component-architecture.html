Rule Engine UI — Component Architecture

Which component structure makes sense? Click to select.

A

Monolith Dashboard

One <alx-rule-engine-dashboard> component that includes everything — tabs for templates, rules, run history, settings. Consumer drops one tag and gets the full UI.

Pros

  • Simplest for consumers — one import, one tag
  • Consistent navigation built-in

Cons

  • All-or-nothing — can't use just the rule editor
  • Hard to customize layout
B

Individual Components

Each component is standalone: <alx-rule-editor>, <alx-template-editor>, <alx-run-history>, etc. Consumer picks what they need and arranges them in their own layout.

Pros

  • Maximum flexibility — use one or all
  • Consumer controls layout
  • Each component is small and focused

Cons

  • Consumer must wire up navigation themselves
  • More imports to manage
C

Both — Dashboard + Individual

Export individual components AND a pre-built dashboard. Consumer can use <alx-rule-engine-dashboard> for the full experience, or import individual components for custom layouts.

Pros

  • Works for both quick setup and custom builds
  • Dashboard is just a composition of the individual components

Cons

  • More code to maintain (dashboard is extra)
